Fix:
I have investigated why the advertisement deletion issue persisted despite previous attempts to fix it. Root Cause Analysis I identified several critical issues in the database permissions (RLS) that were causing deletions to fail silently: 1. Restrictive Database Roles: The advertisements table policies were explicitly restricted to the authenticated database role. In many InsForge/PostgREST environments, authenticated requests might not correctly map to this specific role, causing the policy to ignore the request. Every other table in your project uses the public role and relies on internal ID checks. 2. Silent Failures: The frontend was receiving a 204 No Content status from the database, which usually indicates "Success". However, if Row Level Security (RLS) blocks a delete, PostgreSQL returns this same status even though zero rows were deleted. This made the UI show a "Success" message while the data remained intact. 3. Enum Comparison Robustness: The helper functions for checking roles (is_super_admin, is_admin_of) were comparing ENUM values to strings. While usually fine, this can occasionally fail in complex queries without explicit type casting. 4. RLS Recursion: I found evidence of infinite recursion in the user_profiles and user_neighborhoods tables, which likely caused the permission checks to timeout or return false during the deletion process. Fixes Applied 1. Database Policy Overhaul * Removed Role Restrictions: Updated all advertisements policies to use the public role (matching other project tables) while maintaining strict auth.uid() checks. * Combined Policies: Merged the "Super Admin" and "Neighborhood Admin" policies into a single robust policy, following the successful pattern used by the questions table. * Robust Helper Functions: Updated is_super_admin() and is_admin_of() with explicit type casting (::text) to ensure reliable comparisons across all database contexts. 2. Enhanced Frontend Verification * Verification via .select(): Updated the deleteAd mutation in src/hooks/useAdminAds.ts to use .delete().select(). This forces the database to return the deleted row. * Explicit Error Reporting: The hook now checks if any rows were actually returned. If the database returns success but no rows were deleted (an RLS block), the app will now correctly display: "Permission denied: You do not have authority to delete this advertisement." * Detailed Logging: Added debug logs showing the exact database response status to help diagnose any future edge cases.
